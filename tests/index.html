Executer des tests avec mocha

Mocha est un framework de test Javascript qui fournit les fonctionnalités suivantes :

  • compatibilité Node.js et navigateur ;
  • gestion du code synchrone et asynchrone (callback, promesses, async/await) ;
  • plusieurs syntaxes (appelées interfaces) sont proposées : par défaut, c'est l'interface BDD (Behaviour Driven Development) ;
  • plusieurs formats de rapport (appelées reporters) sont proposés.

Pour définir un test, il suffit d'utiliser la fonction it(). describe() permet de créer des groupes et sous-groupes de tests : 


const assert = require('assert');
// describe() est une fonction qui définit un "groupe" de tests
describe('Array#indexOf', () => {
  // it() est une fonction qui définit un test (pas d'inclusion)
  it('should return -1 when the value is not present', () => {
    // contenu du test : si le test lance une exception, il échoue
    assert.equal(-1, [1,2,3].indexOf(5));
    assert.equal(-1, [1,2,3].indexOf(0));
  })

  it('should return index when the value is present', () => {
    assert.equal(2, [1,2,3].indexOf(3));
    assert.equal(0, [1,2,3].indexOf(1));
    // s'il arrive à la fin, il réussit
  })
})
Dans l'exemple ci-dessus :

  • assert est un module qui fait parti du coeur de Node.js (ci-dessous, nous présenterons la librairie chai qui est fonctionnellement plus riche) ;
  • describe() et it() sond des fonctions globales qui sont définies par mocha (il n'y a donc pas besoin d'utiliser require pour importer mocha) ;
  • l'intérêt d'utiliser la syntaxe BDD est qu'elle permet de faire des phrases expliquant le cas de test (ex : "it should return -1 when the value is not present")

Il possible de tester une fonction asynchrone. Pour cela, il suffit de rajouter un callback comme paramètre de la fonction, et de l'appeler une fois que toutes les assertions ont été exécutées : 


const assert = require('assert');
const fs = require('fs');
const path = require('path');
// describe() est une fonction qui définit un "groupe" de tests
describe('fs#stat', () => {
  // it() est une fonction qui définit un test (pas d'inclusion)
  it.only('should call callback with size in object', (done) => {
    // contenu du test : si le test lance une exception, il échoue
    fs.stat(path.join(__dirname, 'package.json'), (err, stat) => {
      assert.ifError(err);
      assert.ok(stat);
      assert.ok(stat.size);
      done();
    })
  })
})

Il également possible :

  • d'ignorer un test ou groupe de tests en remplaçant it ou describe respectivement par it.skip ou describe.skip ;
  • de n'exécuter qu'un test ou groupe de tests en remplaçant it ou describe respectivement par it.only ou describe.only ;
  • de rajouter un "hook" before qui s'exécutera avant les tests du groupe de tests dans lequel il se situe ;
  • de rajouter un "hook" beforeEach qui s'exécutera avant chaque test du groupe de tests dans lequel il se situe ;
  • de rajouter un "hook" after qui s'exécutera après les tests du groupe de tests dans lequel il se situe ;
  • de rajouter un "hook" afterEach qui s'exécutera après chaque test du groupe de tests dans lequel il se situe.

describe('Array', () => {
  let array = []

  beforeEach((done) => {
    array = [1, 2, 3];
    done();
  })

  it('should splice() remove elements', {skip: true}, (done) => {
    expect(array.splice(1, 1)).to.deep.equals([1, 3]);
    done();
  })
})
info
Mocha s'exécute en 2 phases:
  • dans un premier temps, il exécute le code des describe et stocke en mémoire tous les tests et les "hooks" qu'il devra exécuter ;
  • dans un second temps, il exécute les tests.
Il faut donc bien faire attention à ne pas initialiser les données de tests en dehors des "hooks" ou des tests.

Pour lancer les tests, il ne reste plus qu'à lancer la commande mocha. Pour cela, il faut :

  • soit avoir installé mocha en devDependencies, et utiliser mocha dans le script test du fichier package.json (recommandé);
  • soit avoir installé mocha de manière globale avec npm install -g mocha.
info
Pour accélerer le développement, il est possible d'utiliser la commande mocha -w, qui relance automatiquement les tests dès qu'un fichier .js est modifié.

Calculer la couverture de code avec Istanbul/NYC

NYC/Istanbul est un outil permettant de calculer la couverture de code. Pour cela, il faut :

  • installer nyc (comme devDependencies ou de manière globale)
  • préfixer la commande mocha par nyc (ex: nyc mocha)

Ecrire des assertions avec chai

Chai permet de faire des assertions plus poussées que le module assert de Node.js.
Tout comme mocha, chai propose plusieurs syntaxes : nous utiliserons expect. 


// notez l'inclusion de chai
const expect = require('chai').expect;

describe('Array#indexOf', () => {
  it('should return -1 when the value is not present', (done) => {
    const array = [1, 2, 3];

    // permet de faire la même chose...
    expect(array.indexOf(5)).to.equal(-1);

    //...et des assertions bien plus expressives !
    expect(array).to.be.an.instanceof(Array).and.to.have.a.lengthOf(3);
    expect(array).to.include(2);
    expect(array).not.to.contain(4);
    done();
  });
});

Il est possible de chaîner plusieurs mots-clés :

  • certains déclenchent une validation : equal() pour vérifier l'égalité, include() pour vérifier qu'un tableau contient un élément...
  • d'autres permettent de modifier une validation : not pour inverser la conditition, deep pour comparer en profondeur (et non pas en référence)
  • certains mots-clés sont sans effet, et ne servent qu'à la compréhension : to, be, an...

Les assertions les plus utilisées :

  • égalité : expect(x)/*not.*/.to./*deep.*/equals(y)
  • existance : expect(x).to.be.null, expect(x).to.exist
  • taille des tableaux : expect(x).to.be.empty, expect(x).to.have.length(y)
  • objets : expect(x).to.have./*nested.*/property('model.name').that.equals(z)
  • contenu : expect(x).to.contain/*.keys*/(y)
  • nombres : expect(x).to.be.at.least(y).and.at.most(z)
  • fourre-tout : expect(1).to.satisfy((num) => {return num > 0;})
  • exception : expect(() => {}).to.throw(/message/)
