NPM : the Node Package Manager
Qu'est-ce que NPM ?

NPM est le gestionnaire de paquets offciel de Node.js et se décompose en 2 outils :

  • un registre en ligne npmjs.com qui héberge de nombreuses librairies
  • un outil en ligne de commande (npm) qui permet de gérer les dépendances de notre projet

Dans l'écosystème JavaScript, la tendance est plutôt d'avoir des librairies simples, qui répondent à un besoin précis. Par exemple :
  • moment : permet de manipuler des dates
  • lodash : permet de manipuler des objets, tableaux...
  • request : permet d'effectuer des requêtes HTTP
  • et pleins d'autres
info
Il existe plus de 700 000 librairies sur NPM, et parmi elles, beaucoup ne sont plus maintenues. Le site NPM affiche le nombre de téléchargements et la date de dernière publication, ce qui permet de choisir entre plusieurs librairies.

Le fichier package.json

Le fichier package.json se situe à la racine du projet et décrit le module.

{
  "name": "nom-du-projet",
  "version": "0.3.1",
  "description": "Elle sera publiée",
  "author": "You ",
  "license": "MIT",
  "repository": "https://github.com/you/nom-du-projet.git",
  "private": true,
  "dependencies": {
    "lodash": "~4.17.11"
  },
  "devDependencies": {
    "mocha": "~5.2.0"
  },
  "bundledDependencies": ["lodash"],
  "main": "./lib/entry_point.js",
  "bin": {
    "start-project": "./bin/start"
  },
  "scripts": {
    "test": "mocha test"
  },
  "engines": {"node": ">=6.0.0"}
}
  • name et version sont obligatoires
  • description, author, license et repository sont présents à titre d'information
  • private permet de définir le module comme étant privé, c'est à dire que NPM refusera de le publier
  • les dépendances sont déclarées à l'aide des champs
    • dependencies : ce sont les librairies dont le code a besoin pour fonctionner en production
    • devDependencies : ce sont les librairies qui ne sont utiles que pour le développement (pour les tests, pour le packaging...)
    • peerDependencies : ce sont les librairies dont le code a besoin, mais qui doivent être déclarées par les applications utilisant notre projet (par exemple, le plugin d'une librairie qui a besoin que cette librairie soit présente)
    • bundledDependencies : c'est la liste des librairies qui doivent être embarquées dans notre package lors qu'il est publié sur le registre NPM
    • optionalDependencies : ce sont les librairies dont l'installation peut échouer (par exemple : l'installation d'une librairie nécessaire uniquement pour Linux peut échouer sur Windows)
  • main : fichier qui est chargé par require('npm-du-projet')
  • bin : permet de déclarer une application exécutable en ligne de commande
  • scripts : permet de définir une commande qui sera executée via npm run <command>
  • engines : permet de définir les versions de Node.js compatibles avec ce projet
Le fichier package-lock.json

Ce fichier a été introduit à partir de la version 5 de NPM. Il contient l'arbre complet des dépendances (avec la version exacte des librairies).

Lors d'un npm install:

  • si le fichier est présent : les dépendances sont installées, en respectant les versions définies dans ce fichier ;
  • sinon : les dépendances sont installées en se basant sur le fichier package.json, et ce fichier est créé.

Ce fichier doit être committé car il permet de reproduire à l'identique un npm install.

Ce fichier a aussi l'avantage d'accélerer le npm install

Déclarer ses dépendances

Au sein du fichier package.json, chaque dépendance est définie par un nom de package ainsi qu'une contrainte sur la version.

Pour la contrainte sur la version, NPM respecte le Semantic Versioning (a.k.a. SemVer) : c'est-à-dire que chaque version s'écrit sour la forme X.Y.Z avec

  • X correspond à une version majeure : ce chiffre est incrémenté quand la nouvelle version contient une incompatibilité par rapport à la version précédente. Il peut donc être nécessaire d'adapater notre code pour qu'il continue de fonctionner avec cette nouvelle version.
  • Y correspond à une version mineure : ce chiffre est incrémenté quand la nouvelle version contient de nouvelles fonctionnalités, sans qu'il soit nécessaire pour nous de modifier notre code.
  • Z correspond à une version patch : ce chiffre est incrémenté quand la nouvelle version contient uniquement des corrections de bugs.

NPM offre plusieurs syntaxes pour déclarer la version d'une dépendance :

  • La syntaxe x.y.z permet de fixer une version précise de la dépendance
  • La syntaxe ^x.y.z permet de fixer le premier chiffre différent de 0 (les chiffres suivants sont au minimum):
    • ^1.2.3 équivaut à >= 1.2.3 <2.0.0
    • ^0.1.2 équivaut à >= 0.1.2 <0.2.0
    • ^0.0.1 équivaut à 0.0.1
  • La syntaxe ~x.y.z permet de fixer x et y (z est au minimum):
    • ~1.2.3 équivaut à >= 1.2.3 <1.3.0
    • ~0.1.2 équivaut à >= 0.1.2 <0.2.0
  • Un dépôt git : git://github.com/user/project.git#commit
  • Un fichier tar.gz : http://bitbucket.org/user/repo?format=tar.gz
  • Il existe d'autres syntaxes moins courantes : https://docs.npmjs.com/files/package.json#dependencies

info
Il est recommandé d'utiliser la syntaxe ^1.2.3 afin d'utiliser automatiquement les dernières versions mineures des librairies. La mise à jour vers une nouvelle version majeure restera manuelle car elle peut nécessiter de modifier le code.
info
Il est recommandé d'utiliser la commande npm install <la-librairie> pour ajouter une librairie : NPM installera le nouveau package et se chargera de mettre à jour les fichiers package.json et package-lock.json.

NPM en ligne de commande

NPM fournit un outil en ligne de commande qui permet de gérer le cycle de vie d'un module :

  • npm init : permet d'initialiser un module NPM (création du fichier package.json
  • npm install : permet d'installer les dépendances dans le dossier node_modules
  • npm install <nom-d-une-librairie> : permet d'ajouter le module nom-d-une-librairie comme dépendance du projet (ajoute le module dans le dossier node_modules et met à jour les fichierspackage.json et package-lock.json)
  • npm install -D <nom-d-une-librairie> : permet d'ajouter le module nom-d-une-librairie dans les devDependencies
  • npm test : permet de lancer les tests (commande définie dans le fichier package.json)
  • npm start : permet de lancer l'application (commande définie dans le fichier package.json)
  • npm run <script> : permet de lancer le script défini dans le fichier package.json
  • npm publish: permet de publier le module sur le registry NPM
info
La commande npm install installe de manière récursive les dépendances des dépendances. C'est pour cela que le dossier node_modules contient souvent plusieurs centaines de modules.
info
Il est possible de définir des scripts qui vont se lancer avant ou après le lancement des scripts. Par exemple, la commande npm install va lancer dans l'ordre les scripts:
  • preinstall
  • install
  • postinstall
Ces scripts sont décrits ici : https://docs.npmjs.com/misc/scripts
info
Certains modules sont utilisables en ligne de commande. Pour les utiliser :
  • si le module est présent dans le dossier node_modules, la commande peut-être appelée dans une balise scripts du package.json ou en utilisant la commande npx
  • sinon, il est possible de l'installer de manière globale à l'OS : npm install -g <nom-d-une-librairie>
